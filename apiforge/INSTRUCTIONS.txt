========================================
  APIForge - Local Postman-like API Client
  Electron + Node.js  |  Windows
========================================

WHAT THIS APP DOES
------------------
A locally-running desktop API client (like Postman) for testing HTTP/REST
endpoints. Nothing leaves your machine; all data is stored locally.

Features included:
  - Request methods: GET, POST, PUT, PATCH, DELETE, HEAD, OPTIONS
  - URL bar with query-param editor (auto key/value rows)
  - Custom request headers (enable/disable per row)
  - Request body: JSON, Form URL-Encoded, Raw (+ JSON "Beautify")
  - Auth helpers: Bearer Token, Basic Auth, API Key header
  - Response viewer: status code, time (ms), size (KB)
  - JSON syntax highlighting + raw/headers view + Copy button
  - Gzip / deflate / brotli response decompression handled automatically
  - Multiple request tabs open at once
  - Collections: save requests, organize, reopen, delete
  - History: last 100 sent requests, click to reload
  - Environments + variables: use {{varName}} anywhere in URL/headers/body/auth
  - All data persisted locally to disk (survives restarts)


PREREQUISITES
-------------
1. Node.js (v18 or newer) installed on Windows.
   Check by opening Command Prompt / PowerShell and running:
       node --version
       npm --version
   If missing, download the LTS installer from https://nodejs.org


HOW TO RUN
----------
1. Unzip "apiforge.zip" to a folder, e.g.  C:\Users\<you>\apiforge

2. Open PowerShell or Command Prompt in that folder:
       cd C:\Users\<you>\apiforge

3. Install dependencies (one time only - downloads Electron, ~200MB):
       npm install

4. Start the app:
       npm start

   The APIForge window should open.


HOW TO TEST AN API ENDPOINT
---------------------------
1. In the method dropdown pick GET (or any method).
2. Type a URL in the address bar. Try a free test endpoint:
       https://jsonplaceholder.typicode.com/todos/1
3. Click "Send".
4. The response (with status, time, size) appears below, JSON highlighted.

Try a POST:
   - Method: POST
   - URL:    https://jsonplaceholder.typicode.com/posts
   - Body tab -> select "JSON" -> paste:
         { "title": "hello", "body": "world", "userId": 1 }
   - Send.

Using environment variables:
   - Click the gear (⚙) next to the environment dropdown.
   - Add an environment, e.g. name "Local", variable key "base"
     value "https://jsonplaceholder.typicode.com".
   - Close the modal, select "Local" in the dropdown.
   - Now use {{base}}/todos/1 in the URL bar.

Saving requests:
   - Click "Save" on any request to store it in a Collection
     (sidebar, left). Click a saved request to reopen it in a new tab.



REGRESSION TESTING (NEW)
------------------------
Goal: after a code change, re-test all your endpoints in seconds and see
PASS/FAIL per endpoint.

Concept: each saved request can have a "baseline" - a known-good response
saved to disk. Running a test re-sends the request and compares the live
response to the baseline. Match = PASS, difference = FAIL (with a field diff).

One-time setup:
  The first time you save a baseline you'll be asked to pick a folder where
  baseline files are stored (one .json per request). You only do this once.

Per endpoint:
  1. Open/save a request to a collection (Save button, top-right).
  2. Send it once and confirm the response is correct.
  3. Open the "Tests" tab -> click "Save Current Response as Baseline".
  4. (Optional) In "Ignore these fields", list fields that change every call
     so they don't cause false failures. Examples:
        id, createdAt, token, user.sessionId, items[*].timestamp, *.updatedAt
     Wildcards:  *  matches any one segment;  [*]  matches any array index.
     Leave empty for a strict exact match.
  5. "Run This Test" checks just this endpoint.

Run ALL (the regression sweep):
  In the sidebar, each collection that has baselines shows a green "Test"
  button next to its name. Click it to run every baseline-backed request in
  that collection, one after another. You get a live progress bar and a
  summary like "8 passed, 2 failed of 10", with an expandable field-level
  diff for each failure.

Typical workflow for a team:
  - Build & save your 10 endpoints, save a baseline for each (green dot shows
    which requests have one).
  - Make code changes to your API.
  - Click the collection's "Test" button -> instantly see which endpoints
    still pass and which broke, and exactly which fields changed.

Baseline files are plain JSON in the folder you chose, so they can be
committed to git or shared with teammates.



PERFORMANCE / LOAD TESTING (NEW)
--------------------------------
Goal: simulate many parallel users hitting your endpoints to see how your
API behaves under load - throughput, response times, percentiles, and
errors - then export a professional PDF report. This mirrors the
performance-testing feature in Postman's paid plans.

How it works:
  Each "virtual user" (VU) runs through all the requests in a collection in
  a repeating loop. Many VUs run in parallel to create real load. All
  requests are sent locally from your machine.

The performance screen mirrors Postman's performance tab feature-for-feature.

To run a performance test:
  1. In the sidebar, click the orange "Perf" button on any collection.
  2. Configure the test:
       - Load Profile (Fixed / Ramp Up / Spike / Peak), Virtual Users,
         and Test Duration in MINUTES.
       - A live preview graph shows the exact virtual-user shape over time
         and a plain-English description, updating as you change settings
         (matches Postman's setup screen).
       - Profile-specific field: Ramp Up shows "Initial load" (VUs to start
         with before ramping); Spike and Peak show "Base load" (the low VU
         level between/around the surge).
       - Data File (optional): pick a CSV or JSON file to inject different
         variable values per virtual user. Use {{columnName}} in any request
         URL, header, or body and each VU substitutes its assigned row.
         Choose Ordered (VU n uses row n) or Randomized mapping.
       - Pass test if: metric (avg/p90/p95/p99/error%/req-per-sec) +
         condition (below/above) + threshold -> sets PASS/FAIL.
       - Run Sequence: the list of requests each virtual user will run, in
         order, looping. DRAG rows to reorder, and UNCHECK any request to
         skip it (just like Postman).
  3. Click "Run Performance Test".

While it runs (Summary tab), you get:
  - A header bar: collection, environment, VUs, profile, start time,
    elapsed/duration, and total requests sent.
  - One combined real-time graph with FOUR lines: virtual users,
    requests/sec, response time, and error %. Click any item in the legend
    below the graph to show/hide that line (toggleable, like Postman).
  - A "Response time" dropdown to switch the response-time line between
    Avg / Min / Max / p99 / p95 / p90.
  - A "Request" dropdown to filter the graph to a single request instead of
    all requests combined (request drill-down).
  - A "Stop" button to end the run early, and "Run Again" to reconfigure.

When the run finishes:
  - Summary tab: PASS/FAIL verdict, metric cards (total requests, req/sec,
    avg, error rate, min, max, p90, p95, p99), the graph, and a
    "Metrics for Each Request" table (sorted slowest-first).
  - Errors tab: error classes (Timeout, Connection refused, DNS, TLS,
    Connection reset, HTTP 4xx/5xx) each with a count. Click a class to
    expand it and see which requests hit that error and how often.

Export the PDF report:
  Click "Download PDF Report". The PDF matches Postman's report contents:
  config + PASS/FAIL verdict, summary metrics, response-time & throughput
  charts over time, Slowest Requests, Requests With Most Errors,
  Metrics for Each Request, and the Errors breakdown by class.


NOTE ON ACCURACY (warm-up): Before measurement begins, APIForge sends one
uncounted "warm-up" request per endpoint. This absorbs one-time cold-start
costs (DNS lookup, TCP/TLS handshake, server JIT warm-up) that would
otherwise appear as a single huge Max value and make the report look wrong
(e.g. Max 2000ms while P99 is 30ms). With warm-up, Min/Max/percentiles all
reflect steady-state performance.

Metrics explained:
  - Requests/sec (throughput): how many requests your API served per second.
  - Avg response time: average over all successful requests, in ms.
  - p90 / p95 / p99: 90th/95th/99th percentile response times - e.g. p95 =
    95% of requests were faster than this. Percentiles reveal slow outliers
    that an average hides.
  - Error rate: percentage of requests that returned a non-2xx status or
    failed to connect (timeouts, TLS/connection errors).

Export the PDF report:
  After the run, click "Download PDF Report". The PDF includes the config,
  the PASS/FAIL verdict, all summary metrics, the four metric-over-time
  charts, the per-request breakdown, and the error list - ready to share
  with stakeholders.

Tips for a fresher team:
  - Start small (5-10 VUs, 15-30s) to get a feel, then increase.
  - Use a staging/test environment, NOT production, for heavy load.
  - Higher VUs use more CPU/network on YOUR machine; if your laptop is the
    bottleneck the numbers reflect that, not just the API.
  - Use the regression "Test" button to verify correctness, and "Perf" to
    verify speed under load - they complement each other.

PDF TEST REPORT (NEW)
---------------------
After running a collection's tests (the green "Test" button), the results
window shows a "Download PDF Report" button. Click it to save a clean,
shareable PDF containing:
  - A header with collection name, environment, run time, and duration
  - An overall PASS/FAIL verdict
  - Summary cards: total / passed / failed / skipped / pass-rate
  - A per-endpoint results table (status, method, URL, response time)
  - A "Failure Details" section with the exact field-level differences
    (expected vs actual) for every failed endpoint
You choose where to save the .pdf; it can be emailed or attached as-is.
No extra software is needed - the PDF is produced by the app itself.

WHERE YOUR DATA LIVES
---------------------
Collections, history, and environments are saved to:
   C:\Users\<you>\AppData\Roaming\apiforge\apiforge-data\store.json
(Delete that file to reset everything.)
Baseline JSON files live in the folder you pick on first baseline save.


PROJECT STRUCTURE (for adding features later)
---------------------------------------------
   package.json              - app metadata + scripts
   src/main/main.js          - Electron main process; the HTTP request
                               engine (Node http/https) + local storage IPC
   src/main/preload.js       - secure bridge exposing window.api to the UI
   src/renderer/index.html   - app shell / layout
   src/renderer/styles.css   - all styling (dark theme)
   src/renderer/renderer.js  - all UI logic, tabs, collections, env vars,
                               request building, response rendering

Common extension points:
   - New auth type:       authEditor() + buildHeaders() in renderer.js
   - New body type:       bodyEditor() + buildBody() in renderer.js
   - Request scripting / tests: hook after sendRequest() in renderer.js
   - Import/export:       add IPC handlers in main.js + buttons in renderer.js
   - Actual request send happens in doRequest() in main.js

TROUBLESHOOTING
---------------
- "electron not recognized" / app won't start:
      run  npm install  again inside the project folder.
- Blank window: run  npm start -- --dev  to open DevTools and check console.
- HTTPS sites with self-signed certs may fail; that's expected for safety.
